# Memory 实现方案（长时 + 短时）

## 1. 目标与范围

本方案用于对话系统的记忆模块设计，分为两部分：

- 短时记忆：保存最近 5 轮对话，直接用于上下文拼接。
- 长时记忆：
  - 结构化用户画像（JSON，预算不超过 800 tokens）。
  - 每 50 轮对话触发一次摘要压缩，写入用户专属向量库，支持后续“回头检索”。

本方案仅定义产品与工程实现规范，不包含代码。

---

## 2. 核心概念与统一口径

- **轮（turn）**：一次用户输入 + 一次助手回复，记为 1 轮。
- **会话计数**：按用户维度累计（不是单页面会话）。
- **用户隔离**：每个用户的短时缓存、画像 JSON、向量库都必须隔离。
- **时间编号**：压缩窗口使用递增编号（如 `W0001`、`W0002`），并记录时间范围。

---

## 3. 短时记忆设计（5 轮上下文）

## 3.1 功能目标

- 始终保留最近 5 轮完整问答，用于当前轮 Prompt 构造。
- 不做复杂抽取，不做向量化，强调低延迟和稳定性。

## 3.2 数据结构建议

- `user_id`
- `turn_id`
- `timestamp`
- `user_message`
- `assistant_message`
- `intent`（可选，便于后续分析）

采用固定长度队列（长度 = 5），新数据进入时自动淘汰最旧轮次。

## 3.3 注入上下文策略

- 默认将最近 5 轮按时间顺序注入上下文工程。
- 若不足 5 轮，则注入已有全部轮次。
- 注入前可做轻量清洗（去掉系统噪声、重复空白、无意义符号）。

## 3.4 失败兜底

- 若短时缓存读取失败：降级为“仅当前用户问题”继续回答，不阻塞主流程。
- 若写入失败：记录日志并报警，但不影响本轮回复输出。

---

## 4. 长时记忆设计（一）：用户画像 JSON（<= 800 tokens）

## 4.1 目标

持续维护一份紧凑、可读、可控预算的用户画像，支持个性化回答。

## 4.2 建议字段（可扩展）

- `basic_profile`
  - `occupation`（职业）
  - `domain_level`（领域熟悉度）
- `interaction_habits`
  - `language_preference`（语言偏好）
  - `response_style_preference`（偏好简洁/详细）
  - `format_preference`（偏好表格/步骤/示例）
- `special_requirements`
  - `hard_constraints`（如“必须中文”“不要代码”）
  - `soft_preferences`（如“优先给命令”）
- `topic_memory`
  - 最近高频主题
  - 近期项目上下文关键词
- `last_updated_at`
- `confidence`（字段置信度，便于后续清洗）

## 4.3 更新策略

- 每轮对话后执行“增量更新”，只更新有新证据的字段。
- 优先保留高置信度与近期证据，避免被单次异常对话污染。
- 对互斥偏好（如“要详细”与“要简短”）采用“时间近 + 频次高”原则决策。

## 4.4 Token 预算控制（<= 800）

- 预算上限：800 tokens（硬约束）。
- 超预算时的压缩优先级：
  1. 删除冗余示例与重复表述。
  2. 合并同义偏好项。
  3. 仅保留“当前仍有效”的偏好与限制。
  4. 主题记忆仅保留 Top-N 高频项。

## 4.5 使用策略

- 画像只注入与当前问题相关的字段，避免无关信息污染回答。
- 硬约束（如语言要求）优先级高于普通偏好。

---

## 5. 长时记忆设计（二）：每 50 轮压缩入向量库

## 5.1 触发条件

- 每位用户累计每满 50 轮，对最近 50 轮执行一次压缩摘要并入库。
- 触发点：`turn_count % 50 == 0`。

## 5.2 压缩内容建议

每个压缩窗口至少包含：

- 时间范围（起止时间）
- 窗口编号（如 `W0007`）
- 核心问题主题（Top 3~5）
- 关键决策/偏好变化
- 已确认事实（高置信）
- 未解决事项（可选）

摘要要可检索、可追溯、可用于后续回忆，不要写成泛泛总结。

## 5.3 向量库组织

- 按 `user_id` 隔离（用户专属 collection 或命名空间）。
- 每条向量记录 metadata 至少包含：
  - `user_id`
  - `window_id`
  - `start_turn_id`
  - `end_turn_id`
  - `created_at`
  - `time_range`

## 5.4 5 窗口上限与淘汰策略

- 每用户最多保留 5 个压缩窗口。
- 当新增第 6 个窗口时，删除最旧窗口（FIFO）。
- 删除对象包括：向量记录 + 对应索引元数据，防止脏引用。

## 5.5 检索策略（回头检索）

- 在用户提问前/意图识别后执行一次按需检索：
  - 若问题包含“之前说过”“上次那个”“继续上个方案”等回溯信号，强制检索。
  - 否则可按相似度阈值触发。
- 召回 1~2 个最相关窗口，拼接摘要片段进入上下文。
- 若无命中：明确“未检索到历史相关窗口”，继续常规回答。

---

## 6. 端到端流程（建议）

1. 接收用户输入并生成当前轮回复。  
2. 写入短时记忆队列（保留最近 5 轮）。  
3. 增量更新用户画像 JSON。  
4. 若达到 50 轮阈值，执行窗口压缩并写入向量库。  
5. 若超过 5 个窗口，淘汰最旧窗口。  
6. 下轮请求到来时，先读取短时记忆 + 按需检索长时窗口，再组装上下文。  

---

## 7. 质量与监控指标

- **短时命中率**：当前回复引用最近 5 轮信息的成功率。
- **画像有效率**：画像字段被实际采用且用户满意的比例。
- **长时召回准确率**：回头检索窗口是否真正相关。
- **上下文开销**：记忆注入后 token 消耗是否稳定可控。
- **异常率**：写入失败、压缩失败、检索超时比例。

建议至少记录：触发次数、成功率、平均时延、淘汰次数、用户级错误分布。

---

## 8. 风险与查漏补缺建议

- **用户身份漂移风险**：必须确保 `user_id` 稳定，否则会串记忆。
- **偏好过期风险**：旧偏好可能失效，需基于时间衰减。
- **错误记忆固化风险**：低置信信息不要进入“已确认事实”。
- **检索污染风险**：严格用户隔离，禁止跨用户召回。
- **预算失控风险**：JSON 与检索片段都需要 token 上限守卫。

---

## 9. 交付标准（验收口径）

- 可稳定保存并读取最近 5 轮上下文。  
- 用户画像 JSON 可持续更新且长期保持 <= 800 tokens。  
- 每 50 轮自动生成 1 个压缩窗口并入用户向量库。  
- 每用户窗口数超过 5 时自动淘汰最旧窗口。  
- 支持基于历史窗口的回头检索，并可被上下文工程消费。
