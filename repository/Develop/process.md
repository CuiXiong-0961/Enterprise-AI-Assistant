## 文档切分与入库流程（RAG-corpus → 向量数据库）

本文记录“将 `RAG-corpus` 下的 Markdown 文档切分并写入 Chroma 向量库”的实现逻辑与约定，目标是：**可检索、可溯源、可按元数据过滤、可做父子索引扩展上下文**。

---

### 1. 数据源约定（RAG-corpus）

- **存放位置**：所有待入库文档统一放在 `RAG-corpus/` 下的业务子目录（如 `Development/`、`Bugs/` 等）。
- **文件粒度**：**每个文档对应一个 `.md` 文件**（长文档不再拆多个文件，切分在入库阶段完成）。
- **命名约定**：文档所在文件夹以 `作者-文件名-时间` 形式命名，例如：`张三-护理培训-20250301/`。
  - **作者**：用于溯源与过滤。
  - **文件名**：业务主题或文档标题。
  - **时间**：建议 `YYYYMMDD` 或 `YYYY-MM-DD`，用于版本/时效查询与“以最新为准”策略。

> 备注：若存在历史资料不满足命名约定，需要在入库前做一次“元数据回填/规范化”以保证可检索一致性。

---

### 2. 入库目标与集合（Collection）选择

入库前需先确定文档属于哪个业务域，以写入对应 Collection（见 `repository/schema.md`）：

- **培训/设计/开发资料**：写入 `knowledge_docs`
- **行政条例制度**：写入 `policies`
- （本流程面向通用 md 文档，不覆盖 C++ 头文件与 bug 的专用解析链路）

**统一要求**：
- 每条向量记录必须能回溯到源文件（`file_name`/`source_uri` 至少其一）。
- 长文档必须采用**父子 Chunk 索引**，支撑“先定位章节再精读”和上下文扩展。

---

### 3. 元数据抽取（入库前解析）

针对每个 `.md` 文件，入库前先构建一份“文档级元数据”（同文件所有 Chunk 共享）：

- **file_name**：md 文件相对路径（相对 `RAG-corpus/`），用于展示与回溯
- **source_uri**：可选；若有原始来源（URL、网盘、PDF 路径）可填
- **author**：从文件夹名 `作者-文件名-时间` 中解析得到
- **record_date**：从文件夹名中解析得到；转换为可比较的日期字符串（建议 ISO-8601）
- **topic / title**：从文件夹“文件名”部分与 md 首个标题综合生成
- **domain**：若同一 collection 下需要区分（如 `knowledge_docs` 中 training/design），可写入 metadata 便于过滤

同时抽取“结构级元数据”（用于每个 Chunk）：

- **section_title**：当前 Chunk 所属小标题（来自 Markdown 标题层级，如 `##/###`）
- **chunk_index**：同一文档内顺序编号
- **parent_chunk_id**：父子索引用（见下一节）

> 关键点：作者、时间、文件名、小标题等信息应尽量放在 metadata，避免污染 embedding 文本，但需要展示时可拼装输出。

---

### 4. 切分策略：父子索引 + 语义相似度（<= 300 tokens）

#### 4.1 为什么要“父子索引”

长文档直接按固定长度切分会导致：
- 章节语义被切碎，检索结果难以理解；
- 上下文扩展困难（只拿到一小段，缺失背景）。

父子索引做法：
- **父 Chunk（Parent）**：粒度更大（章节/小节级），用于“定位”和“扩展上下文”。
- **子 Chunk（Child）**：粒度更小（段落/语义块），用于“精确召回”。

检索时通常先召回子 Chunk，再根据 `parent_chunk_id` 拉取父 Chunk 或相邻子 Chunk，形成可读上下文。

#### 4.2 切分总流程（推荐）

1. **结构切分（粗粒度）**  
   - 先按 Markdown 标题层级（`#`/`##`/`###`）把文档拆成“候选块”。  
   - 每个候选块附带 `section_title`（标题文本），并记录层级与顺序。

2. **语义相似度二次切分（细粒度）**  
   - 对候选块内部再按段落/句群形成初始片段。  
   - 使用 LangChain 的**语义相似度**方法对相邻片段做“是否应合并/是否应断开”的判定，确保每个最终子 Chunk 内语义更一致。  
   - 核心目标：把“表面连续但主题切换”的位置断开，把“同一主题被换行打散”的位置合并。

3. **长度约束（硬上限）**  
   - 每个子 Chunk 的最终长度不得超过 **300 tokens**。  
   - 超过时按句子边界继续拆分，避免截断在半句/半段。

4. **父 Chunk 构建**  
   - 每个 `section_title` 生成一个父 Chunk：可以是该标题下所有子 Chunk 的拼接摘要，或“标题 + 前导段 + 关键要点”。  
   - 父 Chunk 也应控制长度，避免过长影响 embedding；父 Chunk 的主要用途是“补上下文”，不是“替代全文”。

#### 4.3 父子 ID 与顺序建议

- **doc_id**：同一文件固定（可用相对路径或其 hash）
- **chunk_id**：全局唯一（建议 `doc_id + chunk_index` 或 hash）
- **parent_chunk_id**：
  - 父 Chunk：`null`
  - 子 Chunk：指向所属父 Chunk 的 `chunk_id`
- **chunk_index**：按文档顺序递增，保证可回放

---

### 5. 向量库写入内容与字段（便于查询/展示/过滤）

#### 5.1 向量文本（document）

- 子 Chunk 的向量文本建议包含：
  - `section_title`（作为第一行或前缀）
  - 该子 Chunk 的正文
- 父 Chunk 的向量文本建议包含：
  - 标题层级路径（如 `章节 > 小节`）
  - 本节摘要/要点（用于扩展上下文）

#### 5.2 metadata 字段（建议）

除 `schema.md` 通用字段外，建议补充以下字段以满足你提出的“文件名、作者、时间、小标题”查询诉求：

- `file_name`：源 md 相对路径
- `author`：作者（来自文件夹）
- `record_date`：日期（来自文件夹）
- `title` / `topic`：文档主题
- `section_title`：小标题（用于展示与过滤）
- `data_source`：如 `RAG-corpus/Development`
- `doc_id` / `chunk_id` / `parent_chunk_id` / `chunk_index`：父子索引与顺序

> 说明：Chroma metadata 只支持标量；数组/复杂结构应 JSON 字符串化（与 `repository/models.py: metadata_to_chroma` 约定一致）。

---

### 6. 查询侧使用建议（检索 + 上下文扩展）

1. **先检索子 Chunk**（更精确）  
2. 对命中的子 Chunk：
   - 取其 `parent_chunk_id` 拉取父 Chunk（补背景）  
   - 按 `chunk_index` 拉取相邻子 Chunk（补上下文连续性）  
3. 组装给 LLM 的上下文时：
   - 必须包含来源（`file_name` / `source_uri`）  
   - 若文档有版本/日期，回答中优先提示“以 `record_date` 最新者为准”  

---

### 7. 质量控制与异常兜底

- **切分质量**：
  - 300 tokens 上限是硬约束；宁可多切几段，也不要超长导致 embedding 或 LLM 输入爆炸。
  - 标题缺失的文档：以“段落 + 语义相似度”形成伪小节，`section_title` 可用“未命名小节-序号”。
- **元数据质量**：
  - 文件夹命名不合规：作者/时间解析失败时必须写入 `unknown` 并记录待修复列表。
- **表格/图片**：
  - 表格建议保留原样（markdown 或 html），但子 Chunk 不要包含超大 base64 资源；
  - 若 md 含 `data:image;base64`，建议在入库前做资源提取与链接化（见 `tools/md_stitcher.py` 的处理思路）。

---

### 8. 最小验收标准

- 任意一个 md 文件入库后，能通过语义问题检索到相关子 Chunk；
- 检索结果能展示：**作者、时间、文件名、小标题**；
- 能通过 `parent_chunk_id` 找到父 Chunk 扩展上下文；
- 单个子 Chunk 不超过 **300 tokens**；
- 每条结果能回溯到源文件并可人工核对。
